# Verify

구현이 실제로 동작한다는 구체적인 증거를 수집한다.

## Goal

막연한 "잘 될 것 같다"를 구체적인 증거로 전환한다.

## Workflow

1. 증명해야 할 정확한 동작을 식별한다.
2. 기존 테스트를 먼저 활용한다.
3. 커버리지가 부족하면 가장 좁은 범위의 직접 검증 명령을 실행한다.
4. 자동화로 충분하지 않으면 수동 검증 단계를 기술하고 관찰 가능한 증거를 수집한다.
5. 실제로 검증된 것만 보고한다.

## 검증 순서

1. 기존 테스트 (Unity EditMode / PlayMode 테스트)
2. 빌드 / 컴파일 체크 (C# 컴파일 에러 없음 확인)
3. 좁은 범위의 직접 명령 검증
4. 수동 / 에디터 인터랙티브 검증

## Unity 검증 방법

- **컴파일**: Unity 콘솔에 컴파일 에러가 없어야 한다
- **EditMode 테스트**: `Test Runner > EditMode` 에서 관련 테스트 통과 확인
- **PlayMode 테스트**: 런타임 동작이 필요한 경우 PlayMode 테스트 실행
- **에디터 검증**: Play 모드 진입 후 Inspector/콘솔에서 예상 동작 확인
- **Unity CLI**: Unity CLI Connector가 연결된 경우 빌드/테스트 원격 실행 가능

## Rules

- 증거 없이 변경이 완료되었다고 말하지 않는다.
- 검증 실패 시 실패 내용을 명확히 포함한다.
- 현실적인 검증 경로가 없으면 그렇다고 명시한다. 거짓 통과를 보고하지 않는다.
- 노이즈 많은 전체 로그보다 간결한 증거 요약을 제공한다.

## Output

- 검증한 내용
- 실행한 명령/테스트
- 통과한 항목
- 실패하거나 미검증된 항목

# Deep Interview

막연한 아이디어를 크리스탈 클리어한 명세로 전환하는 소크라테스식 인터뷰 시스템.

수학적 모호성 점수를 사용해 명확성을 측정하고, 기본 임계값(20%) 이하로 내려갈 때까지 대화를 계속한다.

## 사용 시점

- 막연한 아이디어를 실행하기 전 철저한 요구사항 수집이 필요할 때
- "대신 생각해줘", "다 알아서 해줘", "뭘 만들어야 할지 모르겠어" 같은 상황
- 복잡한 기능이라 바로 코딩하면 범위 발견에 낭비가 클 때
- "그게 내가 원한 게 아닌데" 결과를 방지하고 싶을 때

## 사용 안 할 때

- 파일 경로, 함수명, 수락 기준이 이미 구체적인 요청
- 단순한 수정 또는 단일 변경
- 사용자가 "그냥 해줘", "질문 건너뛰어"라고 하면 존중

---

## Phase 1: 초기화

1. 사용자의 아이디어를 파악한다
2. **Brownfield vs Greenfield 판별**:
   - 현재 프로젝트에 관련 기존 코드가 있으면 → **Brownfield**
   - 완전히 새로운 기능이면 → **Greenfield**
3. Brownfield라면: 관련 코드베이스 영역을 먼저 탐색한다 (Glob/Grep/Read 사용)
4. 인터뷰를 시작하며 다음을 알린다:

```
심층 인터뷰를 시작합니다. 실행 전 아이디어를 완전히 이해하기 위해 질문합니다.
각 답변 후 명확성 점수를 보여드립니다. 모호성이 20% 이하로 내려가면 진행합니다.

아이디어: "{초기_아이디어}"
프로젝트 타입: {greenfield|brownfield}
현재 모호성: 100%
```

---

## Phase 2: 인터뷰 루프

`모호성 ≤ 임계값(20%)` 또는 사용자가 조기 종료할 때까지 반복:

### 2a: 다음 질문 생성

- **가장 낮은 명확성 차원**을 타겟으로 한다
- 해당 차원이 왜 현재 병목인지 한 문장으로 설명한다
- 전제를 가정이 아닌 드러내는 질문을 생성한다
- 여러 질문을 한 번에 묻지 않는다 — **라운드당 하나의 질문만**

**차원별 질문 스타일:**
| 차원 | 스타일 | 예시 |
|------|--------|------|
| 목표 명확성 | "정확히 무슨 일이 일어나나?" | "처형 가능하다고 했는데, 처형 조건이 정확히 무엇인가?" |
| 제약 명확성 | "경계선이 어디인가?" | "기존 무기 시스템과 통합되어야 하나, 독립적이어야 하나?" |
| 성공 기준 | "어떻게 되면 완성인가?" | "완성된 결과물을 보고 '맞다'고 할 때 어떤 모습인가?" |
| 맥락 명확성 (brownfield) | "기존 코드와 어떻게 맞나?" | "PlayerController.cs의 기존 입력 처리 방식을 확장하는가, 대체하는가?" |

### 2b: 질문하기

```
라운드 {n} | 타겟: {가장_약한_차원} | 이유: {한_문장_이유} | 모호성: {점수}%

{질문}
```

### 2c: 모호성 점수 계산

답변을 받은 후 모든 차원의 명확성을 0.0~1.0으로 평가:

**Greenfield:**
`모호성 = 1 - (목표 × 0.40 + 제약 × 0.30 + 성공기준 × 0.30)`

**Brownfield:**
`모호성 = 1 - (목표 × 0.35 + 제약 × 0.25 + 성공기준 × 0.25 + 맥락 × 0.15)`

각 차원 평가:
- score: 0.0~1.0
- gap: 아직 불명확한 것 (score < 0.9인 경우)
- weakest_dimension: 가장 낮은 차원

### 2d: 진행 상황 보고

```
라운드 {n} 완료.

| 차원 | 점수 | 가중치 | 가중 점수 | Gap |
|------|------|--------|----------|-----|
| 목표 | {s} | {w} | {s*w} | {gap 또는 "명확"} |
| 제약 | {s} | {w} | {s*w} | {gap 또는 "명확"} |
| 성공 기준 | {s} | {w} | {s*w} | {gap 또는 "명확"} |
| 맥락 (brownfield) | {s} | {w} | {s*w} | {gap 또는 "명확"} |
| **모호성** | | | **{점수}%** | |

다음 타겟: {가장_약한_차원} — {이유}

{점수 ≤ 임계값 ? "명확성 임계값 달성! 진행 가능합니다." : "{가장_약한_차원}에 집중합니다"}
```

## Phase 3: 챌린지 에이전트

특정 라운드에서 관점을 전환:

**라운드 4+: 반론 모드**
핵심 가정에 도전: "만약 그 반대라면?" 또는 "그 제약이 실제로 존재하지 않는다면?"

**라운드 6+: 단순화 모드**
복잡성 제거 탐색: "여전히 가치 있는 가장 단순한 버전은?" 또는 "이 제약 중 실제 필요한 것과 가정된 것?"

**라운드 8+ (모호성 > 0.3):** 핵심 개념 탐색: "이게 진짜로 무엇인가?" 또는 "핵심 개체와 지원 개체를 구분하면?"

각 모드는 한 번만 사용하고 일반 소크라테스 방식으로 복귀.

---

## Phase 4: 명세 결정화

`모호성 ≤ 임계값` 또는 조기 종료 시:

전체 인터뷰 트랜스크립트를 기반으로 명세를 생성하고 보고한다:

```markdown
# Deep Interview 명세: {제목}

## 메타데이터
- 라운드: {count}
- 최종 모호성 점수: {score}%
- 타입: greenfield | brownfield

## 목표
{인터뷰에서 도출된 명확한 목표 설명}

## 제약
- {제약 1}

## 비목표 (명시적으로 제외된 범위)
- {제외 항목 1}

## 수락 기준
- [ ] {테스트 가능한 기준 1}
- [ ] {테스트 가능한 기준 2}

## 노출된 가정
| 가정 | 도전 방식 | 결정 |
|------|----------|------|
| {가정} | {어떻게 질문했나} | {무엇이 결정되었나} |

## 기술 맥락
{brownfield: 탐색 에이전트가 발견한 관련 코드베이스 사항}
{greenfield: 기술 선택 및 제약}

## 명확성 요약
| 차원 | 최종 점수 |
|------|----------|
| 목표 | {s} |
| 제약 | {s} |
| 성공 기준 | {s} |
| 맥락 (brownfield) | {s} |
| **모호성** | **{score}%** |
```

---

## 종료 조건

- **라운드 3+**: 사용자가 "됐어", "시작해", "그냥 해줘"라고 하면 조기 종료 허용 (단, 모호성이 높으면 경고)
- **라운드 10**: 소프트 경고 — "10라운드가 되었습니다. 현재 모호성: {score}%. 계속할까요?"
- **라운드 20**: 하드 캡 — 현재 명확성으로 진행

---

## 조기 종료 경고 형식

```
현재 모호성 {score}% (임계값: 20%). 여전히 불명확한 영역:
  - {차원}: {gap}

계속하면 재작업이 필요할 수 있습니다. 그래도 진행할까요?
```

---

## 규칙

- 한 번에 하나의 질문만
- 가장 약한 차원을 타겟으로, 그 이유를 명시
- Brownfield라면 사용자에게 묻기 전에 코드베이스를 먼저 탐색
- 매 라운드 후 모호성 점수를 투명하게 표시
- 모호성이 임계값 이하가 될 때까지 진행하지 않음
- 명세 작성 후 실행 방향을 사용자에게 확인

# Learner Skill

디버깅이나 구현 과정에서 발견한 프로젝트 특화 인사이트를 `.claude/skills/` 에 스킬로 저장한다.

재사용 가능한 스킬은 복붙할 코드 조각이 아니라, **문제 유형을 어떻게 생각해야 하는지** 가르치는 원칙과 의사결정 휴리스틱이다.

---

## 품질 게이트

스킬을 추출하기 전에 세 가지 모두 true여야 한다:

- "5분 구글링으로 알 수 있나?" → **NO**
- "이 코드베이스에 특화된 내용인가?" → **YES**
- "실제로 디버깅/조사에 상당한 노력이 들었나?" → **YES**

## 인식 신호

다음 상황 후에만 추출한다:

- 깊은 조사가 필요한 까다로운 버그를 해결했을 때
- 이 코드베이스에 특화된 비직관적인 우회법을 발견했을 때
- 잊으면 시간을 낭비하게 되는 숨겨진 함정을 발견했을 때
- 이 프로젝트에 영향을 미치는 문서화되지 않은 동작을 발견했을 때

## 유용한 스킬의 요건

1. **구글 불가**: 검색으로 쉽게 찾을 수 없는 것
   - BAD: "C#에서 null 체크하는 법" ❌
   - GOOD: "UnityEngine.Object에는 == null이 실제 C# null이 아닐 수 있다. is null이나 == null 연산자를 사용해야 하며 ?. 연산자는 사용하면 안 된다" ✓

2. **맥락 특화**: 이 코드베이스의 실제 파일, 에러 메시지, 패턴을 참조
   - BAD: "null 처리를 잘 해라" ❌
   - GOOD: "PlayerController.cs의 _weaponSlot은 Awake에서 초기화되지만 SceneLoader가 additive 로드 시 Start보다 늦게 실행되면 NullRef 발생 — WeaponSystem.cs:42" ✓

3. **정밀한 행동 지침**: 정확히 무엇을 어디서 해야 하는지
   - BAD: "엣지 케이스를 처리해라" ❌
   - GOOD: "Animator.SetTrigger 호출 전 animator.isActiveAndEnabled 체크 필요 — 비활성화된 오브젝트에서 호출 시 콘솔 에러 없이 애니메이션 무시됨" ✓

## Anti-Patterns (추출하지 말 것)

- 범용 프로그래밍 패턴
- 리팩토링 기법
- 라이브러리/Unity API 사용법
- 주니어 개발자가 5분 만에 구글링할 수 있는 것

---

## 워크플로우

### Step 1: 필요 정보 수집

- **문제 설명**: 발생한 구체적인 에러/증상 (실제 에러 메시지, 파일 경로, 라인 번호 포함)
- **해결책**: 일반적인 조언이 아닌 정확한 수정 방법 (코드 스니펫, 파일 경로, 설정 변경)
- **트리거**: 이 문제를 다시 만났을 때 나타날 키워드 (에러 메시지 조각, 파일명, 증상)

### Step 2: 품질 검증

다음 중 하나라도 해당되면 추출하지 않는다:
- 파일 경로, 라인 번호, 구체적인 에러 메시지가 없이 너무 일반적
- 쉽게 구글링 가능 (표준 패턴, Unity API 사용법)
- 코드 스니펫 없이 해결책이 모호
- 너무 광범위한 트리거 키워드

### Step 3: 저장 위치

- **프로젝트 스킬**: `.claude/skills/{topic}/SKILL.md` — 기본값. 이 프로젝트에 특화된 인사이트.

### 스킬 본문 템플릿

```markdown
# [스킬 이름]

## The Insight
발견한 핵심 원칙은 무엇인가? 코드가 아닌 멘탈 모델.

## Why This Matters
이것을 모르면 무엇이 잘못되는가? 어떤 증상이 여기로 이끌었는가?

## Recognition Pattern
이 스킬이 적용되는 시점을 어떻게 아는가? 어떤 신호가 있는가?

## The Approach
코드만이 아닌 의사결정 휴리스틱. Claude가 이 문제를 어떻게 생각해야 하는가?

## Example (선택)
원칙의 예시 — 복붙 재료가 아닌 원칙의 설명으로서.
```

**핵심**: 스킬은 동일한 상황이 아닌 새로운 상황에도 적용 가능해야 재사용 가능하다.
